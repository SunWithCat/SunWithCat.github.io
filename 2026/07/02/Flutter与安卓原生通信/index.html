🔧 核心概念速查

  • flutterEngine（Flutter 引擎）：运行在安卓后台的”Flutter 大脑”，负责渲染 UI、管理内存以及运行 Dart 代码。
  • dartExecutor（Dart 执行器）：引擎中专门负责执行 Dart 代码的机构（可以理解为运行 Dart 的虚拟机入口）。
  • binaryMessenger（二进制信使）：最核心的通信工具，负责在 Dart 和 Kotlin 之间传递底层的二进制数据（Byte）。

🤔 为什么叫 binaryMessenger（二进制信使）？

Flutter 和安卓原生运行在两个不同的运行环境中 —— Kotlin 无法直接读取 Dart 的内存对象，Dart 也无法直接调用 Kotlin 的类。

它们之间唯一的交流方式，就像寄快递一样：

步骤阶段说明
① 打包📦 序列化Flutter 把想发送的数据（比如字符串 "getBatteryLevel"）序列化成一串 0 和 1 的二进制字节流（Binary Bytes）
② 运输🚚 跨边界传递binaryMessenger 抬着这串字节流，跨越 C++ 底层边界，送到安卓原生这一侧
③ 拆包📬 反序列化安卓原生的 MethodChannel 收到字节流后，将其反序列化还原成 Kotlin 的 String

💡 一句话总结

无论你创建的是 MethodChannel 还是 EventChannel，它们的构造函数第一个参数永远都需要传入 binaryMessenger。
因为它才是底层真正干苦力、运送数据的”快递员”，而 Channel 只是套在它外面的一层”精装外壳”。

总结与避坑指南

1. 两种通道的核心对比

通道类型通信模式适用场景类比
MethodChannel一问一答（单次请求）获取电量、调用原生系统弹窗、获取系统版本等HTTP 请求
EventChannel单向持续推送（流式传输）监听传感器数据、手机电量变化、蓝牙扫描、定时器等WebSocket / 广播

2. 生产环境避坑指南

在实际的商业项目开发中，使用原生通信需要特别注意以下几点：

  1. 通道命名规范（唯一性）：
    通道名称建议使用反向域名（如 com.yourcompany.app/feature）。如果两个通道重名，App 运行时会直接崩溃或发生严重的数据错乱。
  2. 必须做好内存释放：
    EventChannel 必须在 Flutter 的 dispose() 方法中调用 cancel() 取消订阅。如果忘记取消，即使页面关闭了，原生的定时器或监听器依然会驻留在内存中，导致内存泄漏。
  3. 永远要用 try-catch 包裹调用：
    只要是跨端调用，就有可能因为原生报错、方法未实现等原因抛出 PlatformException。Flutter 端的调用必须用 try-catch 包裹，防止程序崩溃。
  4. 原生端避免阻塞主线程：
    Kotlin 端的 setMethodCallHandler 默认运行在 Android 的 UI 主线程中。如果要在原生做耗时的 C++ 运算或网络请求，必须开辟子线程（如使用 Kotlin 协程），计算完毕后再切回主线程通过 result.success 返回结果，否则会导致界面卡死。
